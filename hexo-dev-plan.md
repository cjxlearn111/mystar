## 5. 主题结构规划

Hexo 主题建议命名为：

```txt
themes/mystar
```

主题内部建议结构如下：

```txt
themes/mystar/
├── _config.yml
├── layout/
│   ├── layout.ejs
│   ├── index.ejs
│   ├── post.ejs
│   ├── archive.ejs
│   ├── tag.ejs
│   ├── category.ejs
│   ├── about.ejs
│   └── partial/
│       ├── head.ejs
│       ├── header.ejs
│       ├── footer.ejs
│       ├── hero.ejs
│       ├── post-card.ejs
│       ├── post-meta.ejs
│       └── related-posts.ejs
├── source/
│   ├── css/
│   ├── js/
│   └── images/
└── package.json
```

## 9. 样式开发策略

## 9.1 样式组织原则

- 全局 token 放在根变量中
- 页面布局样式单独组织
- 组件样式以模块划分
- Markdown 内容样式单独维护

## 9.2 建议样式文件划分

```txt
source/css/
├── main.css
├── tokens.css
├── base.css
├── layout.css
├── components.css
└── prose.css
```

说明：

- `tokens.css`：设计 token
- `base.css`：reset、body、selection 等
- `layout.css`：容器、区块、栅格
- `components.css`：按钮、标签、卡片、导航等
- `prose.css`：文章正文样式

## 10. 内容更新体验优化

为了让你后续更轻松更新，我们建议从一开始就补这些能力：

### 10.1 文章模板

在 `scaffolds/post.md` 中预置统一 front-matter：

- 标题
- 日期
- 描述
- 标签
- 分类
- 封面

### 10.2 统一封面规范

- 尺寸比例统一
- 图片命名统一
- 放置路径统一

### 10.3 命令简化

后续可在 `package.json` 增加：

- `dev`
- `build`
- `clean`
- `new:post`

让你尽量少记命令。

## 11. 验收标准

## 11.1 视觉验收

- 是否延续参考稿的高级感
- 是否保留足够留白
- 是否避免模板感
- 是否保证标题和正文排版质量

## 11.2 功能验收

- 首页能正常展示文章流
- 文章页能正常渲染 Markdown
- 标签、分类、归档可访问
- 页面在移动端不破版

## 11.3 上线验收

- GitHub Actions 构建成功
- GitHub Pages 能访问
- 自定义域名可解析
- HTTPS 正常

## 12. 风险与控制

### 12.1 主要风险

- 主题开发时过早抽象，反而变慢
- 页面先套功能，导致视觉走形
- 文章样式不统一，长文体验下降
- GitHub Pages 配置细节遗漏，导致上线失败

### 12.2 控制策略

- 先做首页和文章页，再抽公共结构
- 先视觉还原，再做增强能力
- 先跑通本地构建，再接自动部署
- 每个阶段都保留最小可用版本

---

## 13. 我们接下来怎么推进

推荐顺序如下：

1. 先确认这份开发计划
2. 初始化 Hexo 项目
3. 建立主题目录与样式架构
4. 先做首页
5. 再做文章页
6. 再补归档与标签分类页
7. 最后接 GitHub Pages 发布

---

## 14. 一句话结论

Hexo 对我们来说不是套模板，而是作为静态站引擎；真正的核心是我们自己开发 `mystar` 主题，并让它既符合审美，又方便后续发文。
