# PRD: 个人独立博客与知识库系统 (MVP阶段)

## 1. 产品概述

### 1.1 产品目标
构建一个支持独立域名访问、拥有极简高级 UI 的个人静态博客与知识库系统，用于承载长期内容输出、知识沉淀与个人品牌展示。

### 1.2 产品定位
这是一个偏内容表达与知识归档的个人网站，不是通用 SaaS，也不是社区产品。第一阶段以“高质量内容展示”和“优秀阅读体验”为核心，优先完成一个可上线、可持续扩展的个人内容阵地。

### 1.3 目标用户

- 创作者本人：负责内容发布、知识整理、个人品牌表达
- 技术/设计圈同行：负责浏览、阅读、感知作品质量与内容体系

### 1.4 核心链路

核心页面设计（Variant / Stitch）
-> 结构化 Prompt
-> Codex 生成代码
-> 本地 VS Code 调试
-> Git 推送
-> Vercel 自动部署上线

---

## 2. MVP 目标

MVP 阶段的目标不是一次性做全，而是尽快完成一个具备以下特征的可上线版本：

- 有清晰品牌气质的首页
- 有良好阅读体验的文章详情页
- 有稳定的内容组织方式
- 有完善的响应式适配
- 有可持续迭代的工程结构

MVP 的成功标准：

- 能通过独立域名访问
- 能发布并展示 MDX 文章
- 首页与文章页达到设计参考的高级感与克制感
- 在手机、平板、桌面端都可稳定浏览
- 后续新增文章时无需重构整站

## 3. 研发能力边界与职责划分

### 3.1 人类产品/设计师（User）

- 负责产品定义与需求拆解
- 负责 UI / UX 视觉设计
- 使用 Variant 和 Stitch 构建组件库与页面原型
- 负责审查代码逻辑、视觉还原度和最终决策

### 3.2 AI 开发助手（Codex）

- 严格根据 PRD、结构化 Prompt 与设计稿执行开发
- 不得在没有明确指示的情况下擅自添加花哨动画
- 不得偏离设计稿增加不必要 UI 组件
- 负责根据设计稿与结构化 Prompt 生成前端代码
- 负责处理环境配置、依赖安装和部署脚本
- 负责响应代码报错并提供精确 Debug 方案

### 3.3 协作原则

- 设计优先，代码服务于设计还原
- 先确认页面结构，再进入组件开发
- 所有新增交互和视觉表达需尽量克制
- 组件抽象必须为后续维护服务，而不是为了抽象而抽象

## 4. 技术栈规范

### 4.1 前端框架

- Next.js（App Router）
- React

### 4.2 样式方案

- Tailwind CSS

### 4.3 UI 组件基础

- shadcn/ui

要求：

- 仅作为基础结构与可访问性参考
- 必须严格按照 Variant / Stitch 的设计稿进行覆盖与定制
- 禁止直接套用开箱即用的黑盒视觉组件

### 4.4 内容解析

- MDX

要求：

- 支持在 Markdown 中嵌套 React 组件
- 支持文章内容结构化组织
- 支持代码块展示与高亮

### 4.5 部署环境

- Vercel

要求：

- 支持持续集成
- `git push` 后自动触发构建
- 自动生成可访问链接
- 后续支持绑定独立域名

## 5. UI / UX 设计规范要求

### 5.1 整体风格

- 极简主义
- 高端质感
- Apple / iOS 级视觉语言体系
- 内容优先、结构克制、信息清晰

### 5.2 关键视觉元素

- 大量使用留白与排版层级区分信息
- 局部组件（如导航栏、悬浮弹窗）引入 Glassmorphism（毛玻璃效果）
- 色彩与排版规范严格对齐 Stitch 中输出的 Design Tokens
- 强调微交互逻辑，如：
  - 平滑但克制的 hover 状态
  - 清晰的焦点状态
  - 稳定的路由切换体验

### 5.3 视觉约束

- 禁止无必要的炫技型动效
- 禁止高饱和、大面积跳色
- 禁止偏离参考稿的通用模板风
- 禁止把产品做成 SaaS 后台视觉

### 5.4 当前参考方向理解

结合 `cankao.html` 与 `stitch_minimalist_ai_knowledge_base/ethereal_journal/DESIGN.md`，当前项目适合的视觉方向为：

- 高级编辑感
- 安静、留白、低干扰
- 长文友好
- 轻量毛玻璃
- Serif 标题 + Sans 正文

## 6. 核心功能模块（P0 需求）

### 6.1 首页（Home）

目标：
建立个人品牌感与内容入口。

包含模块：

- 极简的个人 / 品牌简介模块
- 文章列表流
- 标题、摘要、日期、分类标签展示
- 全局响应式导航栏

首页要求：

- 首屏快速传达站点定位
- 内容入口清晰
- 视觉上具备明显高级感

### 6.2 文章详情页（Post）

目标：
提供沉浸式阅读体验与结构清晰的知识表达。

包含能力：

- 支持渲染 MDX 格式
- 支持代码高亮
- 支持标题层级、引用块、列表等内容格式
- 限制文本最大宽度，保证阅读舒适度

文章页要求：

- 正文排版稳定
- 元信息不过度打扰阅读
- 代码块、引用块、列表样式统一

### 6.3 响应式与跨端适配

必须提供：

- Mobile
- Tablet
- Desktop

要求：

- 优先使用 Tailwind 断点系统
- 所有关键页面在三端均需保证可读性
- 不允许桌面视觉直接粗暴压缩到移动端

## 7. 信息架构（MVP）

MVP 阶段建议页面结构如下：

1. 首页 `/`
2. 文章详情页 `/posts/[slug]`

可选但建议尽快纳入：

3. 文章归档页 `/posts`
4. About 页面 `/about`

说明：

- 第一阶段最核心的是首页和文章详情页
- 如果时间允许，增加归档页会显著提升内容系统完整性

- `tags`

优点：

## 9. 组件与页面开发原则

### 9.1 开发原则

- 先搭页面骨架，再抽公共组件
- 先还原主要视觉结构，再补微交互
- 以可维护性为前提做抽象
- 公共部分优先组件化：Header、Footer、PostCard、Tag、MDX 样式容器

### 9.2 组件来源原则

- shadcn/ui 仅用于底层能力与可访问性参考
- 视觉表现必须按设计稿重做
- 不直接照搬默认 Button、Card、Dialog 风格

### 9.3 样式原则

- 所有样式优先统一到 token 和 class 规范中
- 保持字体、间距、边框、圆角的一致性
- 减少局部“例外样式”堆积

## 10. 开发流程

### 10.1 项目初始化

1. 在本地 VS Code 中初始化项目结构
2. 配置 Next.js App Router
3. 配置 Tailwind 与基础环境
4. 安装并初始化必要依赖

### 10.2 设计还原开发

1. 梳理 Stitch / Variant 产出的页面结构
2. 提炼 Design Tokens
3. 定义基础布局组件
4. 逐页实现 UI

### 10.3 内容系统接入

1. 建立 MDX 内容目录
2. 定义 frontmatter 结构
3. 实现文章列表读取
4. 实现文章详情渲染

### 10.4 调试与修正

1. 本地调试页面布局
2. 修复响应式问题
3. 修复样式偏差
4. 修复 MDX 与代码高亮问题

### 10.5 上线部署

1. 关联 GitHub 远程仓库
2. 推送代码
3. 连接 Vercel
4. 配置自动化构建
5. 获取线上预览地址

## 12. 目录结构建议

建议项目初期结构如下：

```txt
app/
  page.tsx
  posts/
    [slug]/
      page.tsx
components/
  layout/
  posts/
  ui/
content/
  posts/
lib/
  posts.ts
  utils.ts
styles/
```

说明：

- `app` 承载路由页面
- `components` 承载公共组件
- `content/posts` 存放 MDX 文章
- `lib` 放内容读取与工具函数

## 14. 风险与约束

### 14.1 主要风险

- 过度抽象导致开发变慢
- 视觉追求过高导致 MVP 迟迟无法上线
- 组件库默认样式污染最终页面气质
- 响应式适配只顾桌面端，忽视移动端阅读体验

### 14.2 控制策略

- 先做 P0 页面闭环，再做扩展页面
- 先完成“能上线且好看”的版本，再持续打磨
- 所有 UI 组件以设计还原为准
- 每完成一个页面就做移动端检查
